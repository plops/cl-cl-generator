## 1. What This Is (and Is Not)

This project implements a **subset** of Protobuf-like binary message serialization and a **subset** of gRPC-like client-server communication, written entirely in Common Lisp.

It is **not** wire-compatible with Google's `protoc` compiler or with standard gRPC libraries. It uses a **custom TCP framing protocol** and a **Lisp S-expression schema DSL** instead of `.proto` files.

**What it demonstrates:**
- How `cl-cl-generator` can turn a concise data-model DSL into fully working `defstruct`, serialization, deserialization, client stubs, and server infrastructure — with no manual code writing.
- A correct implementation of the **Protobuf varint encoding** (including two's complement negative integers).
- A correct implementation of **Protobuf wire-type-based field skipping** for forward compatibility.
- A clean **multi-threaded TCP server** using `usocket` and `sb-thread`.
- A complete **error propagation model**: server-side Lisp errors surface as typed exceptions on the client.

**Dependencies:**
- [SBCL](http://www.sbcl.org/) (for `sb-thread`)
- [Quicklisp](https://www.quicklisp.org/)
- `:usocket` — portable socket library
- `:flexi-streams` — octet↔string conversion and in-memory streams

## 4. The Schema DSL

The schema is defined in `gen.lisp` as a standard Common Lisp quoted list. There are two kinds of top-level entries: `:message` and `:service`.

```lisp
(defparameter *schema*
  '((:message phone-number
     ((:number :string 1)       ; field-name  type  field-number
      (:type   :int32  2)))

    (:message person
     ((:name   :string           1)
      (:id     :int32            2)
      (:email  :string           3)
      (:phones (:repeated phone-number) 4)))  ; nested repeated message

    (:message address-book
     ((:people (:repeated person) 1)))

    (:message get-people-request
     ((:query :string 1)))

    (:service address-book-service
     ((:add-person  person            address-book)   ; method  req-type  resp-type
      (:get-people  get-people-request address-book)))))
```

### Field Types

| DSL type | Wire type | CL struct slot type | Notes |
|---|---|---|---|
| `:string` | 2 (length-delimited) | `string` | UTF-8 encoded |
| `:int32` | 0 (varint) | `integer` | Two's complement varint |
| `:bool` | 0 (varint) | `boolean` | `nil` → 0, `t` → 1 |
| `:bytes` | 2 (length-delimited) | `(simple-array (unsigned-byte 8) (*))` | Raw byte vector |
| `(:repeated T)` | 2 (length-delimited, unpacked) | `list` | List of T |
| any message name | 2 (length-delimited) | struct | Nested sub-message |

### Field Numbers

Field numbers (1, 2, 3, …) must be **unique within a message** and **stable across versions**. They are baked into the binary encoding — changing them is a breaking change. You may add new field numbers freely; old deserializers will skip unknown ones.

### Service Methods

Each method entry is `(method-name request-type response-type)`. The generator produces:
- A `defgeneric` for the method, which you implement on a subclass.
- A `call-<method-name>` client stub function.
- A dispatch branch inside `dispatch-<service-name>`.

## 6. Binary Wire Format

The binary encoding is a correct subset of the Protobuf binary wire format.

### Tags

Every field on the wire is preceded by a **tag**, which is a varint encoding both the field number and the wire type:

```
tag = (field-number << 3) | wire-type
```

| Wire type | Meaning | Used for |
|---|---|---|
| 0 | Varint | `:int32`, `:bool` |
| 1 | 64-bit fixed | (not used in this schema) |
| 2 | Length-delimited | `:string`, `:bytes`, nested messages, repeated fields |
| 5 | 32-bit fixed | (not used in this schema) |

### Varint Encoding

Varints are a variable-length encoding for integers. Each byte contributes 7 bits of data; the high bit indicates whether more bytes follow:

```
Byte layout:  [continuation-bit][7 data bits]

Example: encode 300 (binary: 1_0010_1100)
  Split into 7-bit groups from LSB: 0101100  0000010
  Byte 1: 1_0101100 = 0xAC  (continuation bit set)
  Byte 2: 0_0000010 = 0x02  (last byte)
```

**Negative integers**: Negative values are first converted to their 64-bit two's complement representation using `(ldb (byte 64 0) value)`, then encoded as a 10-byte varint. On decode, `(logbitp 63 value)` detects the sign bit and converts back to a negative Lisp integer with `(- value (ash 1 64))`.

### Length-Delimited Fields

Strings, bytes, and nested messages are encoded as:
```
[tag varint] [length varint] [N bytes of data]
```

### Repeated Fields (Unpacked)

Each element of a repeated field is encoded independently with its own tag:
```
[tag] [len] [elem-1 bytes]
[tag] [len] [elem-2 bytes]
...
```

The deserializer accumulates elements into a list using `push` then reverses with `nreverse` to preserve order.

### Default Value Elision

Scalar fields at their default value are **not encoded** on the wire:
- `:int32` and `:bool` fields with value `0` / `nil` are omitted.
- `:string` fields equal to `""` are omitted.

This matches protobuf3 semantics and keeps payloads small. When deserializing, any field not present in the byte stream retains the struct's default value (`0`, `""`, `nil`).

## 7. TCP Framing Protocol

Each RPC call uses a single persistent TCP connection. Multiple calls can be made sequentially on the same connection (the server reads in a loop until the connection closes).

### Request Frame

```
Offset  Size    Field
------  ------  -----
0       2       Method name length   (big-endian uint16)
2       N       Method name          (UTF-8, N = method name length)
2+N     4       Payload length       (big-endian uint32)
6+N     M       Serialized request   (M = payload length)
```

### Response Frame — Success

```
Offset  Size    Field
------  ------  -----
0       1       Status byte = 0
1       4       Payload length       (big-endian uint32)
5       M       Serialized response  (M = payload length)
```

### Response Frame — Error

```
Offset  Size    Field
------  ------  -----
0       1       Status byte = 1
1       2       Error message length (big-endian uint16)
3       K       Error message        (UTF-8, K = error message length)
```

The status byte distinguishes success (`0`) from error (`1`). Any other status byte value causes the client to signal an error.

## 8. Error Handling Reference

Error handling has three distinct layers, each with different responsibilities.

### Layer 1 — Server Dispatch (`dispatch-address-book-service`)

The dispatcher wraps the entire routing `cond` in `handler-case`. **Any** CL `error` condition raised by the service implementation — or from an unknown method name — is caught here.

**What happens on error:**
1. The error message string is obtained with `(format nil "~a" e)`.
2. It is UTF-8-encoded into a byte vector.
3. Status byte `1` is written to the output stream.
4. The error message length (uint16) is written.
5. The error message bytes are written.
6. `finish-output` flushes the response.

The server dispatch function **never propagates an error to its caller**. Business-logic errors stay on the wire.

```lisp
;; What the dispatcher sends back when your service method raises an error:
;;   [0x01]                    ; status = error
;;   [0x00] [0x1A]             ; error message length = 26 bytes
;;   [S i m u l a t e d ...]  ; the error message text
```

**Unknown method names** also produce an error response:
```lisp
(t (error "Unknown method: ~a" method-name))
;; → caught by handler-case → serialized as error response
```

### Layer 2 — Client Stubs (`call-add-person`, `call-get-people`)

After sending the request and flushing the stream, the client reads the status byte:

| Status | Action |
|---|---|
| `0` | Read uint32 payload length, read payload bytes, deserialize response, return it |
| `1` | Read uint16 error length, read error bytes, decode UTF-8, call `(error "Remote RPC error: ~a" ...)` |
| any other | Call `(error "Invalid status byte: ~a" status)` |

**The caller receives a standard CL `error` condition** when the server reports a failure. Use `handler-case` or `ignore-errors` to handle it:

```lisp
;; Catching a remote error:
(handler-case
    (call-add-person stream my-request)
  (error (e)
    (format t "RPC failed: ~a~%" e)))
```

The error message text is exactly what the server's `format nil "~a"` produced on the `error` condition, prefixed with `"Remote RPC error: "`.

### Layer 3 — Per-Connection Handler (`handle-address-book-service-client`)

The per-connection handler runs in its own thread and wraps the read loop in `handler-case`. It catches:
- `end-of-file` — the client closed the connection normally.
- Any other I/O error (e.g. connection reset, network timeout).

On any such error, the handler **silently exits its loop** and closes the socket. It does not crash the listener thread or the server process. The client disconnect is not an error from the server's perspective.

```lisp
;; This is what happens when the client disconnects:
(error (e) (declare (ignore e)))  ; swallowed — not a problem
(ignore-errors (usocket:socket-close socket))
```

`ignore-errors` around `socket-close` is necessary because the socket may already be in a closed/reset state when the handler exits, and attempting to close it again would signal another error.

### Layer 4 — Listener Thread (`start-address-book-service-server`)

The listener thread runs inside `unwind-protect`:

```lisp
(unwind-protect
    (loop
      (handler-case
          (let ((client-socket (usocket:socket-accept server-socket)))
            (sb-thread:make-thread ...))
        (error () (return))))   ; any accept error terminates the loop
  (ignore-errors (usocket:socket-close server-socket)))  ; always clean up
```

If `socket-accept` raises an error (e.g. the server socket was closed externally), the listener loop terminates and the `unwind-protect` cleanup closes the server socket. The return value of `start-address-book-service-server` is the server socket object — close it to initiate shutdown.

### Summary Error Table

| Scenario | What happens |
|---|---|
| Service method signals `error` | Caught by dispatcher, sent as status=1 frame, client re-raises as CL `error` |
| Unknown method name | Same as above, message: `"Unknown method: <name>"` |
| Client disconnects mid-request | Per-connection handler swallows I/O error, closes socket |
| `socket-accept` fails | Listener loop terminates, server socket closed |
| Client reads invalid status byte | Client signals `(error "Invalid status byte: ~a" status)` |
| EOF reading varint at frame boundary | Returns `nil` (graceful termination of deserialize loop) |
| EOF in the middle of a varint | Signals `(error "Unexpected EOF in middle of varint")` |
| Unknown wire type in `skip-field` | Signals `(error "Unknown wire type: ~a" wire-type)` |

## 9. Writing a Service Implementation

To provide a real service, subclass `address-book-service` and implement the generic methods:

```lisp
(use-package :protobuf-grpc-example)

(defclass my-address-book (address-book-service)
  ((storage :initform (make-address-book) :accessor svc-storage)))

(defmethod add-person ((svc my-address-book) (req person))
  ;; req is a PERSON struct with slots: name, id, email, phones
  (format t "Adding: ~a~%" (person-name req))
  (push req (address-book-people (svc-storage svc)))
  ;; Must return an ADDRESS-BOOK struct
  (svc-storage svc))

(defmethod get-people ((svc my-address-book) (req get-people-request))
  ;; req is a GET-PEOPLE-REQUEST struct with slot: query
  (let ((q (get-people-request-query req))
        (result (make-address-book)))
    (dolist (p (address-book-people (svc-storage svc)))
      (when (or (string= q "") (search q (person-name p)))
        (push p (address-book-people result))))
    ;; Must return an ADDRESS-BOOK struct
    result))
```

### Error Signaling from Service Methods

To signal an error back to the client, simply call `error` from within the method:

```lisp
(defmethod add-person ((svc my-address-book) (req person))
  (when (string= (person-name req) "")
    (error "Person name must not be empty"))
  (when (< (person-id req) 1)
    (error "Person id must be a positive integer, got ~a" (person-id req)))
  ;; ... normal logic
  )
```

The dispatcher catches this automatically. The client will receive a CL `error` condition with the message `"Remote RPC error: Person name must not be empty"`.

You do **not** need to write any serialization or network code. The dispatcher handles everything.

## 11. Running the Server

```lisp
(use-package :protobuf-grpc-example)

(let* ((svc    (make-instance 'my-address-book))
       (server (start-address-book-service-server svc "0.0.0.0" 7000)))
  (format t "Server started. Press Enter to stop.~%")
  (read-line)
  ;; Shut down by closing the server socket
  (usocket:socket-close server))
```

`start-address-book-service-server` returns immediately. It spawns a background listener thread named `address-book-service-server-listener`. Each accepted client spawns a new thread named `address-book-service-client-handler`.

### Binding Address

- `"127.0.0.1"` — loopback only (local process access)
- `"0.0.0.0"` — all interfaces (accessible from other machines)

### Shutdown

Close the server socket returned by `start-address-book-service-server`. The listener thread detects the closure via an error on `socket-accept` and exits cleanly. In-progress client connections are not forcibly terminated; their handler threads run to completion.

## 12. Complete Working Example

```lisp
;;;; complete-example.lisp

(push (truename "source/") asdf:*central-registry*)
(ql:quickload '(:protobuf-grpc-example :usocket :flexi-streams))

(use-package :protobuf-grpc-example)

;;; 1. Define a service implementation

(defclass my-service (address-book-service)
  ((db :initform (make-address-book) :accessor svc-db)))

(defmethod add-person ((svc my-service) (req person))
  (when (string= (person-name req) "")
    (error "Name must not be empty"))
  (push req (address-book-people (svc-db svc)))
  (svc-db svc))

(defmethod get-people ((svc my-service) (req get-people-request))
  (let ((q      (get-people-request-query req))
        (result (make-address-book)))
    (dolist (p (address-book-people (svc-db svc)))
      (when (or (string= q "") (search q (person-name p)))
        (push p (address-book-people result))))
    result))

;;; 2. Start the server

(defparameter *svc* (make-instance 'my-service))
(defparameter *server* (start-address-book-service-server *svc* "127.0.0.1" 7000))
(sleep 0.3)  ; give the listener thread a moment to bind

;;; 3. Connect a client

(defparameter *sock* (usocket:socket-connect "127.0.0.1" 7000
                                              :element-type '(unsigned-byte 8)))
(defparameter *stream* (usocket:socket-stream *sock*))

;;; 4. Add some people

(call-add-person *stream*
  (make-person :name "Alice" :id 1 :email "alice@example.com"
               :phones (list (make-phone-number :number "+1-555-0100" :type 0))))

(call-add-person *stream*
  (make-person :name "Bob" :id 2 :email "bob@example.com"))

;;; 5. Query

(let ((book (call-get-people *stream* (make-get-people-request :query ""))))
  (format t "All people (~a):~%" (length (address-book-people book)))
  (dolist (p (address-book-people book))
    (format t "  ~a (id=~a)~%" (person-name p) (person-id p))))

;;; 6. Trigger a remote error

(handler-case
    (call-add-person *stream* (make-person :name "" :id 3))
  (error (e)
    (format t "Remote error caught: ~a~%" e)))

;;; 7. Clean up

(usocket:socket-close *sock*)
(usocket:socket-close *server*)
```

**Expected output:**
```
All people (2):
  Bob (id=2)
  Alice (id=1)
Remote error caught: Remote RPC error: Name must not be empty
```

## 14. Design Decisions

### Why code generation?

The entire content of `source/` is produced mechanically from the `*schema*` list in `gen.lisp`. Adding a new field requires editing the schema once; the generator propagates the change to the struct definition, the serializer, the deserializer, the package exports, and (if the field is on a service type) the client stubs and dispatch table.

This is the core premise of `cl-cl-generator`: use Lisp's homoiconicity — the fact that code and data share the same list representation — to generate repetitive boilerplate from compact descriptions.

### Why custom TCP framing and not HTTP/2?

Implementing HTTP/2 correctly requires handling flow control, header compression (HPACK), stream multiplexing, and TLS — none of which add to the pedagogical value of demonstrating code generation. The custom TCP framing is minimal, correct, and easy to trace with a packet capture or `strace`.

### Why `:usocket` and not raw POSIX sockets?

`:usocket` is the standard portable socket library for Common Lisp. It works on SBCL, CCL, ABCL, and ECL. The server starter uses `sb-thread` directly (SBCL-specific) for simplicity; a portable version would use a thread-pool library like `:lparallel`.

### Why one thread per client?

The simplest correct concurrency model for a demonstration. The listener thread accepts connections and hands each one to a new per-client thread. This works correctly for low connection counts. A production system would use a worker thread pool.

### Why `defstruct` instead of `defclass`?

`defstruct` generates typed slot accessors, a typed `make-*` constructor, and `equalp` support automatically. It is more appropriate than `defclass` for plain data transfer objects that have no polymorphic behaviour. Service implementations use `defclass` because they need `defmethod` dispatch.

### Why elide default-value fields?

Omitting zero/empty/nil values from the wire encoding matches protobuf3 semantics and produces smaller payloads. The cost is that you cannot distinguish "field was set to zero" from "field was absent" — but in protobuf3, these are semantically identical.

### Why UTF-8 for strings and error messages?

UTF-8 is the universal encoding for protobuf wire format strings and is self-synchronizing (a truncated byte sequence is detectable). `:flexi-streams` provides `string-to-octets` / `octets-to-string` with `:external-format :utf-8`.

## 15. Extending the Schema

### Adding a New Field to a Message

```lisp
;; In gen.lisp, add to the :person message:
(:message person
 ((:name  :string 1)
  (:id    :int32  2)
  (:email :string 3)
  (:phones (:repeated phone-number) 4)
  (:active :bool  5)))   ; ← new field
```

Re-run `./run_gen.sh`. The struct, serializer, deserializer, and exports are automatically updated.

Old clients reading new messages will see `active = nil` (the default). New clients reading old messages will see `active = nil` (field absent from wire → default). Forward and backward compatible.

## 16. Known Limitations

| Limitation | Reason |
|---|---|
| Not wire-compatible with `protoc` | Uses custom TCP framing; no HTTP/2, no Protobuf file descriptors |
| No stream multiplexing | One request at a time per connection (sequential, not concurrent) |
| No TLS | Raw TCP only |
| SBCL-only threading | Uses `sb-thread`; portable threading requires `:lparallel` or `:bordeaux-threads` |
| No service reflection | No mechanism to discover methods or message types at runtime |
| Wire types 3 and 4 not supported | Deprecated protobuf group fields are treated as unknown wire type (error) |
| No streaming RPCs | Request and response are both single messages; no server-streaming or bidirectional streaming |
| Error messages capped at 65535 bytes | Error message length field is uint16 |
| Max payload size 4 GB | Payload length field is uint32 |
