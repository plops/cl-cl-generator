# Test Instructions: Configurable Gentoo Dockerfile Meta-Generator

We need to verify and test the newly implemented Gentoo Dockerfile meta-generator and its build pipeline.
The codebase is located in:
`/workspace/src/cl-cl-generator/example/05_dockerfile_meta/source01/examples/01_gentoo/`

Note: Here and in the rest of the prompt we refer to /workspace/src/ this folder is actually /home/kiel/stage on the current system

## Key Files to Verify
1. `gen_gentoo.lisp`: Contains the configuration parameters (e.g. `*target-machine*`, `*minimal-image*`, feature/package flags) and generates the Dockerfile using inline heredocs and hybrid Portage caching mounts.
2. `build.sh`: Runs the generator, boots the Docker BuildKit export build, redirects logs to `output/build.log`, and syncs (`rsync` / `cp`) the newly compiled packages and distfiles back to the host directories for future read-only bind mounts.
3. `enter_container.sh`: Allows entering the interactive build stage (`base` target) for debugging.
4. `config/`: Contains the large configuration files (like `config6.18.18` and `dwm-6.8`).

---

## Action Plan

Please execute the following steps systematically:

### Step 1: Run the Docker Build and Output Export
1. Change directory to `/workspace/src/cl-cl-generator/example/05_dockerfile_meta/source01/examples/01_gentoo/`.
2. Check if Docker and BuildKit are enabled on your system:
   `docker info`
3. Execute the build script:
   `./build.sh`
4. This script will:
   - Compile `gen_gentoo.lisp` and generate a fresh `Dockerfile`.
   - Initialize host directories `./distfiles`, `./binpkgs`, and `./output` if they do not exist.
   - Run `docker build -o ./output .` using BuildKit, exporting all compiled files and cache files to the host.
   - Synchronize files from `./output/distfiles` and `./output/binpkgs` back into your host directories `./distfiles` and `./binpkgs`.

### Step 2: Verify Build Outputs on the Host
Verify that the `./output/` directory contains:
- `gentoo.squashfs_e14` (ThinkPad squashfs, with NVIDIA/CUDA stripped out).
- `vmlinuz` (Compiled kernel).
- `initramfs_squash_sda1-x86_64.img` (Dracut initramfs).
- `packages.txt` & `packages.tsv` (Installed packages size and compilation time statistics).
- `packages_obsolete.txt` (Validation report generated by `eix-test-obsolete` to detect stale package flags).

Verify that `./distfiles/` and `./binpkgs/` on your host now contain the files downloaded/compiled during the build.

### Step 3: Verify Caching & Speed (Second Build)
Run the build script a second time:
`./build.sh`
- Verify that the build runs significantly faster and does not re-download distfiles, since `./distfiles` is bind-mounted read-only and will satisfy all package downloads.

### Step 4: Verify Interactive Debugging Environment
1. Execute the interactive enter script:
   `./enter_container.sh`
2. Verify that you get dropped into an interactive bash shell inside the base stage, with your host `distfiles` and `binpkgs` mounted.
3. Exit the container.

Report the validation results, including build times and confirmation of files in `./output`.
